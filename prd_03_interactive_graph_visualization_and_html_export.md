# PRD 03 — Interactive Graph Visualization and HTML Export

## Title

**Interactive Graph Visualization and HTML Export for Runtime Visual Debugging**

## Document Purpose

This PRD defines the third implementation milestone for the Visual Debugging for AI-Generated Python Code project.

This milestone takes the directed execution graph produced in PRD 02 and renders it into an interactive HTML visualization that a human can actually use.

If PRD 01 gives the project a memory and PRD 02 gives it structure, this milestone gives it eyes.

The goal is to produce a graph visualization that is:

- interactive
- legible
- module-aware
- useful for quick comprehension
- rich enough to inspect task metadata
- simple enough to ship fast

This is the first milestone where the product becomes visibly demoable.

---

## Objective

Build a visualization/export layer that converts the NetworkX execution graph into an interactive HTML artifact using PyVis.

The exported graph must make it easy to answer questions like:

- What major modules ran?
- What tasks executed inside each module?
- What order did things happen in?
- Which tasks failed?
- How long did each task take?
- What kind of data flowed through each step?

The visualization must prioritize clarity over flash.

---

## Why This Milestone Comes Third

A correct graph object is not yet a product. It is a skeleton in a Python process.

The conference demo, debugging value, and overall user experience depend on turning that graph into something a human can scan and click.

This milestone comes after graph construction because visualization should reflect real structure, not invent it. Rendering before the data model is settled is how software ends up with beautiful nonsense.

So this milestone focuses on honest visual translation of the graph model already established.

---

## Scope

This milestone includes:

- converting NetworkX graph data into a PyVis network
- exporting the graph as standalone HTML
- rendering nodes with module-aware grouping
- exposing task metadata in node tooltips or details
- mapping task status to node color
- displaying durations as metadata
- displaying input/output previews as metadata
- preserving execution order hints where useful
- making the HTML view usable in a browser on laptop or phone
- enabling a readable demo experience for the synthetic pipeline

This milestone does **not** include:

- FastAPI routes
- file upload UI
- PDF-specific pipeline logic
- LLM-generated summaries
- live graph updates during execution
- async execution views
- advanced custom frontend work
- true nested submodule rendering beyond module → task
- production-grade layout optimization for huge graphs

---

## Inputs and Outputs

### Input

The visualization layer consumes a graph generated by PRD 02.

Expected input:

- `networkx.DiGraph`
- task execution nodes with metadata
- edges representing execution flow
- module metadata available on each node
- status metadata available on each node

### Output

The visualization layer produces:

- a standalone HTML file containing the interactive graph
- optionally a helper function that returns the HTML path
- optionally a helper that returns raw HTML string if useful later

The HTML should be viewable directly in a browser without requiring a separate frontend app.

---

## Product Requirements

### Functional Requirement 1 — Use PyVis for Rendering

The visualization layer must use PyVis to render the graph into interactive HTML.

#### Requirements

- Accept a NetworkX graph as input
- Convert it into a PyVis network
- Export as standalone HTML
- Be easy to open directly in the browser

#### Rationale

PyVis offers a fast path to interactive graph visualization without requiring custom frontend engineering.

---

### Functional Requirement 2 — Render Executed Tasks as Interactive Nodes

Each execution node in the graph must appear as a visual node in the HTML graph.

#### Requirements

Each rendered node must visibly include or expose:

- task name
- module grouping
- status color
- detailed metadata on hover or click

At minimum, the node label should show:

- task name

Tooltip or detail metadata should include:

- task description
- module name
- duration
- execution status
- input preview
- output preview
- data lengths
- error message if present

#### Design principle

The graph should be readable at a glance, but rich on inspection.

Do not dump enormous text into the visible label area. Put details into hover text or tooltip content.

---

### Functional Requirement 3 — Module-Aware Grouping

The visualization must make modules visually distinguishable.

#### Requirements

Nodes belonging to the same module should be grouped using PyVis grouping or visual conventions.

Possible approaches include:

- shared group value
- consistent color family
- spatial clustering encouraged by group metadata

#### Important constraint

This milestone does **not** require true nested visual containers or expandable module boxes.

The MVP requirement is simpler:

A user should be able to tell which tasks belong to which module.

#### Recommendation

Use PyVis `group` based on `module_name`.

This is enough for MVP and avoids layout wizardry turning into a side quest.

---

### Functional Requirement 4 — Status-to-Color Mapping

Task status must be visually obvious.

#### Requirements

At minimum, support color mapping for:

- success → green
- error → red

Optional later support:

- warning / suspicious → yellow

#### Requirements

- Color choice must be deterministic
- Color must derive from node status metadata, not inference
- Failed tasks must stand out clearly
- If a task has no recognized status, fall back to a neutral default

#### Important note

This milestone should not invent validation logic. It only visualizes status already present on the node.

---

### Functional Requirement 5 — Duration Visibility

Duration must be visible in node metadata.

#### Requirements

Duration should appear in tooltip / hover content in a human-readable format.

Examples:

- `4.2 ms`
- `0.81 s`

#### Recommendation

Keep duration out of the main node label unless the graph remains clean. The safer default is tooltip metadata.

---

### Functional Requirement 6 — Data Preview Visibility

Input and output previews must be inspectable in the graph.

#### Requirements

Node tooltip/details should include:

- input preview
- output preview
- approximate input length
- approximate output length

#### Important design constraint

The visualization must remain readable even when preview strings are long.

Recommendations:

- keep visible node labels short
- place previews in hover text / title HTML
- truncate if necessary before rendering

This milestone should visualize the preview summaries already computed upstream, not recompute them.

---

### Functional Requirement 7 — Error Display

If a task failed, the visualization must make that obvious.

#### Requirements

For error nodes:

- use red color
- include error message in tooltip
- preserve duration if available
- preserve input preview if available

A user should be able to quickly find the failure point without reading the source code.

---

### Functional Requirement 8 — Execution Flow Arrows

Directed edges must be visible as arrows or clearly directional flow.

#### Requirements

- edge direction should reflect parent → child execution
- arrows should be enabled if supported by PyVis configuration
- graph should visually communicate flow, not just adjacency

This matters because the whole point is seeing what triggered what.

---

### Functional Requirement 9 — HTML Export Helper

The system must provide a straightforward way to export the rendered graph to an HTML file.

#### Recommended primary interface

Possible function shape:

```python
render_graph_html(graph, output_path="output/graph.html") -> str
```

This should:

- render the graph
- save HTML to disk
- return the output path

Optional helper:

```python
graph_to_html_string(graph) -> str
```

Only include if it helps later FastAPI integration without complicating MVP.

---

### Functional Requirement 10 — Mobile-Friendly Enough for Demo Use

The HTML visualization should be usable from a phone browser well enough for conference testing and on-the-go review.

#### Requirements

- graph loads without custom browser setup
- hover/click interaction remains usable
- exported HTML is self-contained enough for deployment or direct serving
- layout should not completely collapse on small screens

This does not need to be perfect mobile UX. It just needs to be demo-capable.

## Visual Design Guidance

### Design goal

The graph should feel like a software instrument panel, not a chaotic data-art poster.

### Recommended visible node label

Keep it minimal:

- task name only
- optionally short module hint if readability remains good

### Recommended tooltip content

Tooltip content can be HTML-formatted and should include fields like:

- Task: `add_numbers`
- Description: `Add two numbers to compute an intermediate value`
- Module: `Math Operations`
- Status: `success`
- Duration: `4.2 ms`
- Input length: `7`
- Output length: `1`
- Input preview: `length=7, head='(2, 3)', tail='(2, 3)'`
- Output preview: `length=1, head='5', tail='5'`

For error nodes also include:

- Error: `division by zero`

### Module grouping style

Recommended MVP approach:

- use `group=module_name`
- allow PyVis to cluster/group naturally
- preserve module name in tooltip
- optionally prefix label or title if needed

### Color strategy

Recommended mapping:

- success → readable green
- error → readable red
- warning/suspicious → readable yellow if later added
- unknown/default → neutral gray or blue

### Layout philosophy

Readable > clever.

If the layout is slightly boring but understandable, good.
If it looks like a cosmic spaghetti accident, bad.

## Technical Design Guidance

### Recommended file structure for this milestone

```text
graph/
  graph_visualizer.py
```

### Likely responsibilities of `graph_visualizer.py`

Should include:

- conversion from NetworkX graph to PyVis network
- node styling logic
- edge styling logic
- tooltip/title formatting
- HTML export
- optional layout configuration helper

### Suggested primary functions

Likely candidates:

- `render_graph_html(graph, output_path="output/graph.html") -> str`
- `build_pyvis_network(graph) -> Network`
- `make_node_title(node_data: dict) -> str`
- `status_to_color(status: str) -> str`

The exact names can vary, but these responsibilities should remain separated enough to debug cleanly.

---

## Configuration Guidance

### Physics / layout

PyVis defaults can be chaotic. The implementation should choose a conservative, readable configuration.

Recommendations:

- use stable-ish physics settings
- do not over-tune initially
- prefer readability over fancy movement
- if needed, reduce physics intensity after initial stabilization

### Node size

Keep node sizes moderate and mostly uniform for MVP.

Do not size nodes based on payload length or duration yet unless it is trivial and clearly beneficial.

### Fonts and labels

Keep labels readable.
Avoid cramming metadata into labels.
Use tooltip content for detail.

### Edge style

Simple directional edges are enough.
Do not over-style edge thickness, dashes, or animations yet.

---

## Graph Semantics to Preserve

This milestone must not distort the meaning of the underlying graph.

### Preserve:

- one node per execution instance
- one directed edge per parent-child relationship
- module metadata on each node
- status metadata on each node
- execution order metadata where useful

### Do not introduce:

- fake module container nodes
- synthetic summary nodes
- hidden edge rewrites
- guessed relationships not present in graph data

Visualization should reveal the graph, not editorialize it.

---

## Non-Functional Requirements

### Clarity

A user unfamiliar with the code should be able to open the HTML and quickly understand:

- what tasks exist
- which module each belongs to
- what order they flowed in
- where a failure occurred

### Simplicity

The implementation should remain lightweight and debuggable.

Avoid:

- custom frontend complexity
- over-engineered templating
- clever but fragile PyVis hacks

### Extensibility

The visualization layer should leave room for future additions such as:

- module-level summary panels
- LLM graph summaries
- clickable side panels
- PDF upload workflow
- warning/yellow state
- nested module views
- live updates

But do not implement those here unless they are trivial and low-risk.

---

## Acceptance Criteria

This milestone is complete when all of the following are true:

1. A valid NetworkX execution graph can be rendered to standalone HTML
2. Every graph node appears as an interactive visual node
3. Node labels are readable and not overloaded
4. Node tooltip/details expose task description, module, duration, previews, and status
5. Error nodes are visibly distinct from success nodes
6. Directed execution flow is visually represented by edges/arrows
7. Module membership is visually distinguishable enough for a user to notice
8. The exported HTML can be opened in a browser and used without additional setup
9. A developer can inspect the graph visually and match it back to the runtime trace

---

## Definition of Done

This milestone is done when a developer can:

1. run a tiny decorated program,
2. convert its trace into a graph,
3. render that graph as standalone HTML,
4. open the HTML in a browser,
5. and immediately understand the program better by inspecting the visual output.

At that point, the project stops being an internal data pipeline and becomes a visible product.

That is the first real “aha” moment of the whole build.

---

## Implementation Cautions

Do not let visualization complexity eat the project alive.

Avoid:

- hand-building a frontend when PyVis already solves the MVP
- trying to implement zoomable nested module UIs now
- spending hours fine-tuning layout physics before the basic view is working
- overloading node labels with every scrap of metadata
- adding live update behavior in this milestone

This milestone is about getting to a clear, honest, demoable graph — not inventing the Louvre of debugging tools on day one.
